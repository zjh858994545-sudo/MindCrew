## 一、基础概念

### 1.1 MySQL 有哪些存储引擎？InnoDB 和 MyISAM 的区别？

**答**：常见存储引擎：InnoDB（默认）、MyISAM、Memory、Archive、CSV。生产环境基本只用 InnoDB。

| 维度 | InnoDB | MyISAM |
|:---|:---|:---|
| 事务 | 支持 ACID | 不支持 |
| 锁粒度 | 行锁 + 表锁 | 只有表锁 |
| 外键 | 支持 | 不支持 |
| 索引结构 | 聚簇索引（数据 + 索引同一棵 B+ 树） | 非聚簇（数据和索引分离） |
| 崩溃恢复 | 支持（redo log） | 不支持 |
| 全文索引 | 5.6+ 支持 | 支持（早期更强） |
| 并发性能 | 高（行锁） | 低（表锁） |
| 适用场景 | OLTP、写多 | 读多写少、日志类 |

MySQL 5.5+ 默认 InnoDB。MyISAM 的优势（全表 COUNT 快、空间小）在 8.0 时代基本被 InnoDB 追平。

### 1.2 InnoDB 的架构是什么？各组件作用？

**答**：InnoDB 内部分两层。

**内存结构（Buffer Pool）**：

- **Buffer Pool**：缓存数据页和索引页。默认 128MB，生产环境一般设为物理内存的 50%-70%。包含 LRU 链表管理热数据。
- **Change Buffer**：非唯一索引页修改的缓冲。索引页不在 Buffer Pool 时，先把修改记录到 Change Buffer，下次读到这个页时再 merge。
- **Log Buffer**：redo log 缓冲区，默认 16MB。事务提交时按策略刷到磁盘 redo log file。
- **Adaptive Hash Index**：自适应哈希索引，对热点页自动建哈希加速。

**磁盘结构**：

- **System Tablespace**：系统表空间（ibdata1），存数据字典、双写缓冲（Doublewrite Buffer）、Change Buffer 的持久化部分。
- **File-per-table Tablespace**：默认每张表一个 .ibd 文件，存数据 + 索引。
- **Redo Log**：物理日志，循环写。崩溃恢复用。
- **Undo Log**：逻辑日志，存事务前镜像。MVCC + 回滚用。

**后台线程**：Master Thread（主线程，周期刷脏页）、IO Thread、Purge Thread（清理 undo log）、Page Cleaner Thread。

### 1.3 char 和 varchar 的区别？

**答**：

| | char(n) | varchar(n) |
|:---|:---|:---|
| 长度 | 固定 n 字符 | 可变，最大 n |
| 存储 | 不足 n 用空格填充 | 实际长度 + 1-2 字节长度前缀 |
| 检索 | 略快（无需算长度） | 略慢 |
| 末尾空格 | 检索时会去除 | 保留 |
| 适合场景 | 长度固定（手机号、身份证、MD5） | 长度变化大（用户名、内容） |

InnoDB 中 varchar(255) 之内长度前缀用 1 字节，超过用 2 字节。VARCHAR 的 n 是字符数（不是字节数），utf8mb4 字符集下一个字符最多占 4 字节。

### 1.4 datetime 和 timestamp 的区别？

**答**：

| | datetime | timestamp |
|:---|:---|:---|
| 占用 | 8 字节 | 4 字节 |
| 范围 | 1000-01-01 到 9999-12-31 | 1970-01-01 到 2038-01-19（32 位限制） |
| 时区 | 与时区无关，存什么读什么 | 存的是 UTC，读取时按当前会话时区转换 |
| 默认值 | 5.6 之前不支持自动填充 | 一直支持 CURRENT_TIMESTAMP |
| NULL | 真正 NULL | 默认 NULL 时存 0000-00-00 00:00:00 |

实践建议：

- 跨时区应用用 timestamp（自动转时区）
- 单一时区或要存历史日期用 datetime
- 2038 年问题在云原生时代已经被 64 位 OS 解决，新版 MySQL 8 的 timestamp 也支持更大范围

### 1.5 NULL 值对查询和索引的影响？

**答**：

1. **索引仍可存储 NULL**：InnoDB 的 B+ 树索引把 NULL 当作最小值排在最前面。所以 `WHERE col IS NULL` 能走索引，`WHERE col IS NOT NULL` 也能。
2. **不能用 = 比较**：`col = NULL` 永远返回 false，必须用 `IS NULL`。
3. **聚合函数忽略 NULL**：`COUNT(col)` 不包含 NULL，但 `COUNT(*)` 包含。`SUM(col)` 在所有 NULL 时返回 NULL 而非 0。
4. **唯一索引允许多个 NULL**：MySQL 实现里 NULL ≠ NULL，一个唯一索引可以有多个 NULL 值。
5. **影响优化器**：NULL 让优化器统计不准，复合索引中带 NULL 列时 cardinality 估算偏差。
6. **占额外空间**：每行有一个 null bitmap 记录哪些字段是 NULL。

实践建议：业务字段尽量 `NOT NULL` + 默认值（数字用 0 / 字符串用空串），减少歧义。

## 二、索引

### 2.1 索引的底层数据结构是什么？B+ 树和 B 树的区别？

**答**：MySQL InnoDB 用 B+ 树。

**B+ 树特点**：
- 多叉平衡查找树（一般 3-4 层就能存几千万行）
- **非叶子节点只存索引**，不存数据。能在固定页内（16KB）放更多索引项，扇出更大
- **叶子节点存数据**，且叶子之间用双向链表连接，区间查询 / 排序很快

**B 树（B-Tree）**：每个节点既存索引也存数据，扇出小，查询性能不稳定，区间查询要做中序遍历，慢。

**为什么选 B+ 树**：

1. 单节点存索引多 → 树更矮 → 磁盘 IO 少（数据库瓶颈是 IO）
2. 叶子节点链表 → 范围查询 / ORDER BY 高效
3. 查询性能稳定（一定经过 log N 层叶子节点）

**为什么不用红黑树/AVL 树**：内存数据结构，节点太小、树太高，磁盘场景层数太多 IO 爆炸。

**为什么不用哈希**：哈希只支持等值查询，不支持范围、排序、LIKE 前缀。InnoDB 的自适应哈希索引是辅助优化，不是主索引。

### 2.2 聚簇索引和非聚簇索引的区别？

**答**：

- **聚簇索引（Clustered Index）**：叶子节点存整行数据。InnoDB 表必有一个聚簇索引：
  - 若有主键，主键就是聚簇索引
  - 否则取第一个唯一非空索引
  - 都没有的话 InnoDB 自动生成隐藏的 6 字节 row_id

- **非聚簇索引（Secondary Index / 二级索引）**：叶子节点存"索引列值 + 主键值"。要拿完整行还要回表查聚簇索引一次。

**实际影响**：

- 聚簇索引查询快（一次 IO 就拿到完整行）
- 二级索引一般要回表，查询多消耗一次 IO
- 主键过长会让所有二级索引都跟着膨胀（叶子节点都存主键）
- 推荐用 BIGINT AUTO_INCREMENT 做主键（短、有序、写入快）

MyISAM 不一样：MyISAM 所有索引都是非聚簇的，索引和数据完全分离，二级索引和主键索引地位相同。

### 2.3 主键索引、唯一索引、普通索引、联合索引的区别？

**答**：

| | 唯一性 | NULL | 数量 | 用途 |
|:---|:---|:---|:---|:---|
| 主键索引 | 唯一 | 不允许 | 一张表只能一个 | 行的唯一标识 |
| 唯一索引 | 唯一 | 允许（多个 NULL） | 多个 | 业务唯一性约束 |
| 普通索引 | 不唯一 | 允许 | 多个 | 加速查询 |
| 联合索引 | 看是否声明 unique | 允许 | 多个 | 多列组合查询 |

**联合索引（复合索引）**：在多个列上建一个索引，如 `INDEX idx(a, b, c)`。比建三个单列索引更高效，因为：

1. 一次索引扫描搞定多列匹配
2. 二级索引存所有列值 + 主键，能形成覆盖索引
3. 三个单列索引在 WHERE 多列时优化器最多用一个

### 2.4 什么是覆盖索引？什么是回表？

**答**：

**回表（Lookup）**：查二级索引拿到主键，再用主键查聚簇索引取完整行。两次索引扫描。

**覆盖索引（Covering Index）**：查询所需的所有字段都在某个二级索引里，**不需要回表**。EXPLAIN 的 Extra 列显示 `Using index` 表示用了覆盖索引。

例子：

```sql
-- 索引：idx_user_name (user_id, name)
-- 表字段还有 age, email

-- 这条要回表（查 age）
SELECT user_id, name, age FROM t WHERE user_id = 1;

-- 这条覆盖索引（user_id 和 name 都在索引里）
SELECT user_id, name FROM t WHERE user_id = 1;
```

实践技巧：经常一起查询的字段做联合索引，让查询走覆盖索引。

### 2.5 联合索引的最左前缀原则？

**答**：联合索引 `(a, b, c)` 等价于隐式建了 `(a)`、`(a,b)`、`(a,b,c)` 三个索引，**必须从最左列开始连续使用才能命中**。

| 查询条件 | 是否命中 |
|:---|:---:|
| WHERE a=1 | 命中 a |
| WHERE a=1 AND b=2 | 命中 a,b |
| WHERE a=1 AND b=2 AND c=3 | 命中 a,b,c |
| WHERE a=1 AND c=3 | 只命中 a，c 不能用索引 |
| WHERE b=2 AND c=3 | 完全不命中 |
| WHERE a=1 AND b>2 AND c=3 | 命中 a,b，c 不能用索引（范围之后失效）|
| WHERE a=1 ORDER BY b | 命中 a，b 用于排序避免 filesort |

**注意点**：

- 范围查询（`>`、`<`、`BETWEEN`、`LIKE 'xx%'`）之后的列无法使用索引
- 等值查询不影响后续列，范围查询会"截断"后续列的使用
- 优化器有时会自动调整 WHERE 子句顺序，所以 `b=2 AND a=1` 等价于 `a=1 AND b=2`

### 2.6 索引下推（ICP）是什么？

**答**：Index Condition Pushdown，MySQL 5.6+ 默认开启。

**没有 ICP**：

1. 存储引擎层用索引找匹配的行
2. 把所有匹配行返回给 Server 层
3. Server 层用剩余 WHERE 条件过滤

**有 ICP**：在存储引擎层就直接用 WHERE 条件过滤，减少回表次数。

例子：

```sql
-- 索引 idx(name, age)
SELECT * FROM t WHERE name LIKE '张%' AND age > 20;
```

- 无 ICP：找到所有 `name LIKE '张%'` 的行（可能很多），全部回表后过滤 age
- 有 ICP：在二级索引里就用 age 条件过滤，**只对 age > 20 的行回表**

EXPLAIN 的 Extra 显示 `Using index condition` 表示用了 ICP。

### 2.7 什么情况下索引会失效？

**答**：

1. **不符合最左前缀**：联合索引 `(a,b,c)` 用 `WHERE b=2` 直接失效。
2. **范围查询后的列失效**：`WHERE a=1 AND b>2 AND c=3`，c 不走索引。
3. **对索引列做函数 / 运算**：`WHERE DATE(create_time) = '2024-01-01'`、`WHERE id+1 = 10`。
4. **隐式类型转换**：字段 varchar 但查询 `WHERE phone = 13800001234`（数字），MySQL 内部会 `CAST(phone AS UNSIGNED)`，索引失效。反过来字段是数字查字符串也一样。
5. **LIKE 前导通配符**：`WHERE name LIKE '%张'` 或 `'%张%'` 失效。`LIKE '张%'` 能用索引。
6. **使用 OR**：如果 OR 任一侧的列没有索引，整个 OR 不走索引（5.0+ 能 index_merge 时另说）。
7. **NOT IN / != / <>**：通常不走索引（要看选择性）。
8. **优化器判断全表扫描更快**：表很小或预估返回大部分数据时。

排查工具：`EXPLAIN`。如果优化器选错索引可以用 `FORCE INDEX` 强制。

### 2.8 如何判断一条 SQL 有没有用到索引？EXPLAIN 怎么看？

**答**：`EXPLAIN SELECT ...` 输出关键列：

| 列 | 含义 |
|:---|:---|
| id | 查询序列号，越大越先执行 |
| select_type | SIMPLE / PRIMARY / SUBQUERY / DERIVED 等 |
| table | 涉及的表 |
| type | 访问类型，最重要的列。从好到差：`system > const > eq_ref > ref > range > index > ALL` |
| possible_keys | 可能用的索引 |
| **key** | 实际用的索引（NULL 表示没走索引）|
| key_len | 索引使用的字节数（判断联合索引用了几列） |
| ref | 显示哪些列被用于查找 |
| **rows** | 估算扫描行数 |
| filtered | 过滤后剩余比例 |
| **Extra** | 关键附加信息 |

**type 各值**：

- `system`：表只有一行（特殊）
- `const`：通过主键 / 唯一索引等值查询，最多一行
- `eq_ref`：连接时用主键 / 唯一索引，每次匹配一行
- `ref`：普通索引等值查询，可能多行
- `range`：范围查询
- `index`：扫整个索引（不访问数据）
- `ALL`：全表扫描，**最差**

**Extra 关键值**：

- `Using index`：覆盖索引
- `Using where`：用 WHERE 过滤
- `Using index condition`：用了 ICP
- `Using filesort`：额外排序（性能差，应避免）
- `Using temporary`：用临时表（GROUP BY / DISTINCT 时常见，应避免）
- `Using join buffer`：表连接没走索引

### 2.9 前缀索引是什么？什么时候用？

**答**：对长字符串字段（VARCHAR(255) / TEXT），只对前 N 个字符建索引，节省空间。

```sql
CREATE INDEX idx_email ON users(email(20));
```

**选择前缀长度**：

```sql
-- 计算不同前缀长度的选择性，越接近 1 越好
SELECT COUNT(DISTINCT LEFT(email, 10)) / COUNT(*) FROM users;
SELECT COUNT(DISTINCT LEFT(email, 20)) / COUNT(*) FROM users;
```

通常选择性达到 0.9+ 的最小长度。

**限制**：

- 无法用作覆盖索引（需要原始值时还要回表）
- 无法用于 ORDER BY、GROUP BY

### 2.10 索引的选择性是什么？

**答**：选择性 = 不重复值 / 总行数。范围 0-1，越接近 1 越好。

```sql
SELECT COUNT(DISTINCT col) / COUNT(*) FROM t;
```

- 主键选择性 = 1（最好）
- 性别字段选择性 ≈ 0.5（一般差）
- 邮箱字段接近 1（好）

低选择性字段不适合单独建索引（如性别、状态枚举），但可以放联合索引的非首位列。

### 2.11 为什么不建议用太多索引？

**答**：

1. **写入慢**：INSERT / UPDATE / DELETE 都要维护索引。每多一个索引，写性能下降。
2. **占空间**：每个索引都是一棵 B+ 树。
3. **优化器选择慢**：候选索引太多，优化器评估开销大。
4. **碎片**：频繁更新导致索引页空洞，需要定期 OPTIMIZE TABLE。

经验：单表索引数控制在 5-7 个。优先建联合索引覆盖多场景。

### 2.12 什么是自适应哈希索引？

**答**：InnoDB 自动监控热点页，对频繁访问的 B+ 树页自动建立内存哈希索引，把"B+ 树查找"变成"O(1) 哈希查找"。

**特点**：

- 自动管理，无法手动控制
- 只对等值查询有效
- 高并发写场景下争用 latch，可能反而成为瓶颈，可用 `innodb_adaptive_hash_index = OFF` 关闭

## 三、事务

### 3.1 事务的四大特性 ACID 分别是什么？

**答**：

- **A（Atomicity）原子性**：事务里所有操作要么全成功要么全失败回滚。靠 **undo log** 实现。
- **C（Consistency）一致性**：事务前后数据满足业务约束。这是业务层、约束（外键、唯一）、其他三个特性共同保证的结果。
- **I（Isolation）隔离性**：并发事务互不干扰。靠 **锁 + MVCC** 实现。
- **D（Durability）持久性**：提交后数据持久化。靠 **redo log** 实现。

### 3.2 MySQL 的四种隔离级别是什么？

**答**：

| 隔离级别 | 中文 |
|:---|:---|
| READ UNCOMMITTED | 读未提交 |
| READ COMMITTED | 读已提交（Oracle 默认）|
| REPEATABLE READ | 可重复读（**MySQL 默认**）|
| SERIALIZABLE | 串行化 |

设置：`SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;`
查看：`SELECT @@transaction_isolation;`

### 3.3 各隔离级别分别解决了哪些问题？

**答**：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|:---|:---:|:---:|:---:|
| READ UNCOMMITTED | 可能 | 可能 | 可能 |
| READ COMMITTED | 否 | 可能 | 可能 |
| REPEATABLE READ | 否 | 否 | InnoDB 用 MVCC + 临键锁解决 |
| SERIALIZABLE | 否 | 否 | 否 |

**三种问题**：

- **脏读（Dirty Read）**：读到别的事务**未提交**的数据
- **不可重复读（Non-Repeatable Read）**：同一事务内两次读同一行，结果不一样（中间被其他事务 UPDATE）
- **幻读（Phantom Read）**：同一事务内两次范围查询，结果集行数不一样（中间被其他事务 INSERT）

### 3.4 MySQL 默认隔离级别是什么？为什么？

**答**：REPEATABLE READ（可重复读）。

历史原因：MySQL 早期主从复制用的是基于语句（statement-based）的 binlog，READ COMMITTED 下并发事务执行顺序可能导致主从数据不一致。RR + 临键锁能保证语句执行的串行化效果。

现在多用 binlog ROW 格式后，RC 也安全了。Oracle / PostgreSQL 默认都是 RC。但 MySQL 为了兼容历史，依然是 RR。

互联网公司很多手动改成 RC，因为：

- 并发性能更好（间隙锁少）
- 业务上幻读不常见
- 语义更直观

### 3.5 MVCC 的原理是什么？

**答**：MVCC（Multi-Version Concurrency Control）= 多版本并发控制。让读不阻塞写、写不阻塞读，大幅提升并发性能。

**核心数据结构**：

每行隐藏字段：

- `DB_TRX_ID`：最后修改该行的事务 ID（6 字节）
- `DB_ROLL_PTR`：指向 undo log 中前一个版本的指针（7 字节）
- `DB_ROW_ID`：行 ID（若无主键时使用）

通过 `DB_ROLL_PTR` 串起来一个**版本链**，所有历史版本都在 undo log 里。

**ReadView（一致性视图）**：

事务开始时（RC 是每个 SELECT 时，RR 是事务第一个 SELECT 时）生成 ReadView，包含：

- `m_ids`：当前活跃事务 ID 列表
- `min_trx_id`：m_ids 中的最小值
- `max_trx_id`：下一个要分配的事务 ID
- `creator_trx_id`：当前事务 ID

**可见性规则**：对版本链上每一行，判断 `DB_TRX_ID`：

- < min_trx_id：可见（已提交）
- ≥ max_trx_id：不可见（未来事务）
- 在 m_ids 中：不可见（活跃事务，未提交）
- 不在 m_ids 中（且小于 max_trx_id）：可见（已提交）

不可见就沿 `DB_ROLL_PTR` 找历史版本。

### 3.6 undo log、redo log、binlog 分别是什么？有什么区别？

**答**：

| | undo log | redo log | binlog |
|:---|:---|:---|:---|
| 层级 | InnoDB | InnoDB | MySQL Server |
| 内容 | 逻辑日志（前镜像）| 物理日志（页修改）| 逻辑日志（SQL / row） |
| 写入时机 | 修改前 | 修改后 | 事务提交时 |
| 持久化 | 持久化（undo tablespace）| 循环写文件 | 追加写文件 |
| 作用 | 回滚 + MVCC | 崩溃恢复 | 主从复制 + 数据恢复 |
| 大小 | 配 undo tablespace | innodb_log_file_size | max_binlog_size |

### 3.7 事务提交时 redo log 和 binlog 的两阶段提交流程？

**答**：

为什么需要两阶段？保证 redo log 和 binlog 一致——若一个写了一个没写，主从数据 / 崩溃恢复都会出问题。

**流程**：

1. **Prepare 阶段**：InnoDB 写 redo log，状态 prepare
2. 写 binlog 到磁盘
3. **Commit 阶段**：InnoDB 把 redo log 状态改为 commit

**崩溃恢复时**：

- 如果 redo log 是 commit：正常恢复
- 如果 redo log 是 prepare 且 binlog 完整：提交
- 如果 redo log 是 prepare 且 binlog 不完整：回滚

参数：

- `innodb_flush_log_at_trx_commit`：1（每次提交刷盘，最安全）/ 2（写 OS cache）/ 0（每秒刷）
- `sync_binlog`：1（每次提交刷盘）/ 0（OS 决定）

生产推荐 `1, 1`（双 1）配置。

### 3.8 什么是当前读和快照读？

**答**：

- **快照读（Consistent Read）**：基于 MVCC 读历史版本，不加锁。普通 `SELECT` 都是快照读。
- **当前读（Current Read）**：读最新已提交的数据，会加锁。
  - `SELECT ... FOR UPDATE`（X 锁）
  - `SELECT ... LOCK IN SHARE MODE`（S 锁）
  - `INSERT / UPDATE / DELETE`（X 锁）

RR 隔离级别下，事务里多次快照读结果一致；但当前读会读到最新数据（导致同一事务内 SELECT 不同结果是正常的）。

## 四、锁

### 4.1 MySQL 有哪些锁？

**答**：按粒度分类：

- **全局锁**：`FLUSH TABLES WITH READ LOCK`，锁整库。用于全库备份。
- **表锁**：
  - 表级 Share / Exclusive 锁：`LOCK TABLES ... READ/WRITE`
  - MDL（Metadata Lock）：访问表时自动加，DDL 时升级为写锁
  - 意向锁（IS / IX）：在表上声明"我要在表内某些行加 S / X 锁"
- **行锁**（InnoDB）：
  - Record Lock：记录锁
  - Gap Lock：间隙锁
  - Next-Key Lock：临键锁（Record + Gap）
  - 插入意向锁：INSERT 前的特殊 Gap 锁

按模式分类：

- **共享锁（S）**：读锁，互相兼容
- **排他锁（X）**：写锁，互斥
- **意向锁**：表级，标记"内部有行锁"，加速判断"能否加表锁"

### 4.2 行锁、表锁、间隙锁、临键锁分别是什么？

**答**：

- **行锁（Record Lock）**：锁住某一行的索引项。`WHERE id = 1 FOR UPDATE` 锁住主键值为 1 的索引行。
- **表锁**：锁整张表。MyISAM 默认就是表锁；InnoDB 在没有索引可用时会"行锁退化为表锁"。
- **间隙锁（Gap Lock）**：锁一个开区间，防止其他事务在区间内插入。例如索引上有 1, 5, 10，对 5 的间隙锁锁住 (1, 5) 这个区间。
- **临键锁（Next-Key Lock）**：行锁 + 间隙锁，锁住一个左开右闭区间 `(1, 5]`。InnoDB 在 RR 隔离级别下默认加临键锁来防止幻读。

### 4.3 什么情况下会加间隙锁？

**答**：仅在 RR 隔离级别下，且：

1. **范围查询当前读**：`SELECT * FROM t WHERE id > 10 AND id < 20 FOR UPDATE`
2. **等值查询不存在的记录**：`SELECT * FROM t WHERE id = 8 FOR UPDATE`（id=8 不存在，加 (前一个值, 后一个值) 间隙锁）
3. **唯一索引上的范围查询**：哪怕字段是唯一的，范围查询仍可能加间隙锁

RC 隔离级别下没有间隙锁（除外键检查等少数场景）。

### 4.4 死锁是怎么产生的？如何避免？如何排查？

**答**：

**死锁产生条件**：

1. 互斥（资源不能共享）
2. 持有并等待
3. 不可剥夺
4. 循环等待（A 等 B，B 等 A）

典型场景：两个事务用相反顺序更新两行。

**避免**：

1. 固定加锁顺序（所有事务按相同顺序访问资源）
2. 事务尽量短小，减少锁持有时间
3. 把可能冲突的事务拆开（异步化）
4. 用乐观锁代替悲观锁
5. 适当用 `SELECT ... FOR UPDATE NOWAIT` 或 `SKIP LOCKED`

**排查**：

```sql
-- 查看最近死锁
SHOW ENGINE INNODB STATUS;
-- 看 LATEST DETECTED DEADLOCK 段

-- 查看当前锁等待
SELECT * FROM information_schema.INNODB_TRX;
SELECT * FROM performance_schema.data_locks;
SELECT * FROM performance_schema.data_lock_waits;
```

InnoDB 检测到死锁会**自动回滚代价小的事务**（撤销更少行），另一个事务正常完成。

### 4.5 乐观锁和悲观锁的区别及使用场景？

**答**：

- **悲观锁**：先加锁再操作。假设冲突一定会发生。`SELECT ... FOR UPDATE` 是数据库悲观锁。
- **乐观锁**：先操作再检查冲突。假设冲突很少。通过 **版本号** 或 **CAS** 实现。

```sql
-- 乐观锁实现
UPDATE t SET data = ?, version = version + 1 WHERE id = ? AND version = ?;
-- 若影响行数 = 0，说明被别人改过，重试
```

| | 悲观锁 | 乐观锁 |
|:---|:---|:---|
| 冲突频率高 | 适合（一次加锁解决）| 不适合（频繁重试）|
| 冲突频率低 | 浪费（大部分锁是无用的）| 适合 |
| 实现 | DB / Redis 锁 | 版本号 / CAS |
| 性能 | 阻塞，并发差 | 无阻塞，但需重试 |

### 4.6 SELECT FOR UPDATE 和 SELECT LOCK IN SHARE MODE 的区别？

**答**：

| | FOR UPDATE | LOCK IN SHARE MODE |
|:---|:---|:---|
| 锁类型 | X 锁（排他）| S 锁（共享）|
| 别的事务读 | 普通读不阻塞（快照读）| 普通读不阻塞 |
| 别的事务加 S 锁 | 阻塞 | 允许 |
| 别的事务加 X 锁 | 阻塞 | 阻塞 |
| 使用场景 | 接下来要修改 | 只是想确保读到的数据不被改 |

注意：必须在事务里使用（`BEGIN; SELECT ... FOR UPDATE; ... COMMIT;`），否则没意义。

### 4.7 锁等待超时怎么处理？

**答**：

参数 `innodb_lock_wait_timeout` 默认 50 秒。超时会报错 `Lock wait timeout exceeded; try restarting transaction`。

**应对**：

1. **业务层重试**：捕获异常，指数退避重试
2. **缩短大事务**：把长事务拆成多个小事务
3. **优化加锁顺序**：减少死锁概率
4. **提高超时值**：对于已知的长事务（如批处理）适当调高
5. **使用 NOWAIT / SKIP LOCKED**：MySQL 8.0+
   - `FOR UPDATE NOWAIT`：拿不到锁立即报错
   - `FOR UPDATE SKIP LOCKED`：拿不到锁跳过这行，常用于消息队列

## 五、查询优化

### 5.1 SQL 执行顺序是什么？

**答**：书写顺序 vs 执行顺序：

```sql
-- 书写顺序
SELECT DISTINCT col, COUNT(*)
FROM t1 JOIN t2 ON t1.id = t2.id
WHERE t1.status = 1
GROUP BY col
HAVING COUNT(*) > 5
ORDER BY col DESC
LIMIT 10;
```

**执行顺序**：

1. `FROM` + `JOIN ON`：确定数据源，做笛卡尔积 + ON 过滤
2. `WHERE`：行级过滤
3. `GROUP BY`：分组
4. `HAVING`：分组后过滤（这里能用聚合函数）
5. `SELECT`：选择列 + 计算表达式
6. `DISTINCT`：去重
7. `ORDER BY`：排序
8. `LIMIT`：截取

理解执行顺序的意义：知道 WHERE 不能用 SELECT 的别名（WHERE 比 SELECT 早执行），但 ORDER BY 能用。

### 5.2 EXPLAIN 各字段含义？type 字段各值代表什么？

**答**：（详见 2.8 节）

type 从好到差：

- `system`：表只有一行（极少见）
- `const`：主键 / 唯一索引等值查询，最多一行
- `eq_ref`：JOIN 时被驱动表用主键 / 唯一索引等值匹配
- `ref`：普通索引等值查询，可能多行
- `fulltext`：全文索引
- `ref_or_null`：类似 ref，多了 NULL 处理
- `index_merge`：多个索引合并使用
- `range`：范围扫描（`>`、`<`、`IN`、`BETWEEN`）
- `index`：索引全扫描（不访问数据）
- `ALL`：全表扫描，最差

目标：至少到 `range`，理想是 `ref` 或更好。

### 5.3 慢查询如何排查和优化？

**答**：

**排查流程**：

1. 开启慢查询日志：
   ```sql
   SET GLOBAL slow_query_log = ON;
   SET GLOBAL long_query_time = 1;  -- 超过 1 秒算慢
   ```

2. 分析慢日志：用 `mysqldumpslow` 或 `pt-query-digest`

3. 对具体 SQL 跑 `EXPLAIN` / `EXPLAIN ANALYZE`（8.0+）

4. 用 `SHOW PROFILE` 看各阶段耗时

5. 看 `SHOW PROCESSLIST` 找当前阻塞

**优化策略**：

1. 加索引、改索引顺序
2. 避免索引失效（函数、隐式转换、前导通配符）
3. 减少回表（覆盖索引）
4. 优化 SQL（join 顺序、子查询改 join）
5. 分页优化（延迟关联、游标）
6. 拆 SQL（一个大查询拆成多个）
7. 改表结构（冷热分离、归档）

### 5.4 大表分页查询如何优化？

**答**：

`SELECT * FROM t ORDER BY id LIMIT 1000000, 20` 在大表上极慢：需要扫 1000020 行再丢弃前 100 万。

**优化方案**：

**方案 1：游标分页（推荐）**

```sql
-- 第一页
SELECT * FROM t ORDER BY id LIMIT 20;
-- 拿到最后一行 id = N
-- 下一页
SELECT * FROM t WHERE id > N ORDER BY id LIMIT 20;
```

每次都 O(20)，跟页数无关。缺点：只能"下一页"，不支持跳页。

**方案 2：延迟关联**

```sql
SELECT t.* FROM t INNER JOIN (
    SELECT id FROM t ORDER BY id LIMIT 1000000, 20
) AS sub ON t.id = sub.id;
```

子查询只扫主键索引（小），主查询用 id 直查（快）。比直接 SELECT * 快 10 倍以上。

**方案 3：业务限制**

- 限制最大可翻到第 N 页
- 超过 N 引导用筛选条件缩小范围

### 5.5 深度分页问题如何解决？

**答**：本质同 5.4。补充几个工程方案：

- **冗余排序字段**：维护一个连续 sort_id 字段，分页用 WHERE 范围
- **二级缓存**：把 ID 列表缓存到 Redis（zset），分页只从 Redis 取
- **搜索引擎**：ES 的 search_after 也是游标式
- **业务降级**：列表页只展示最近 N 条，历史数据走搜索

### 5.6 联表查询和子查询哪个性能更好？

**答**：

通常 **联表（JOIN）比子查询好**。原因：

- 子查询有时被优化器物化成临时表
- 子查询的关联条件不能利用索引（旧版本）
- JOIN 更利于优化器统一规划执行计划

**但 5.6+ 后 MySQL 对子查询做了 IN-to-EXISTS、半连接优化**，大部分子查询性能跟 JOIN 接近。

**子查询适用场景**：

- 子查询结果集很小，且需要 DISTINCT 语义
- IN 子查询且外表很大（半连接优化）

**JOIN 适用场景**：

- 多表关联且都需要返回字段
- 关联字段都有索引

实际工作中两种都跑 EXPLAIN 对比。

### 5.7 COUNT(\*)、COUNT(1)、COUNT(字段) 的区别？

**答**：

| | 含义 | 是否包含 NULL | 性能 |
|:---|:---|:---|:---|
| COUNT(*) | 行数 | 是 | InnoDB 会选最小的索引扫描 |
| COUNT(1) | 行数（同 *）| 是 | 同 COUNT(*) |
| COUNT(字段) | 该字段非 NULL 行数 | 否 | 要读字段，可能更慢 |

**COUNT(\*) 和 COUNT(1) 性能基本一样**，MySQL 内部都会优化。

**InnoDB 的 COUNT(\*)**：没有快速计数，必须扫描。MySQL 8.0 用并行扫描略快。如果业务对 COUNT 要求高，建议：

- 用 Redis 维护计数器
- 用估算值（`SELECT TABLE_ROWS FROM information_schema.tables WHERE table_name = 't'`）

MyISAM 缓存了行数，COUNT(*) 不带 WHERE 是 O(1)。

### 5.8 IN 和 EXISTS 的区别及使用场景？

**答**：

```sql
-- IN
SELECT * FROM A WHERE A.id IN (SELECT B.a_id FROM B);

-- EXISTS
SELECT * FROM A WHERE EXISTS (SELECT 1 FROM B WHERE B.a_id = A.id);
```

**老经验**：

- IN：外大内小（B 小，先查 B 列表，再 A 比对）
- EXISTS：外小内大（对 A 每行查 B 是否存在）

**MySQL 5.6+ 优化器智能化后**：两者大部分情况性能一致，优化器会自动选最好的执行方式。

实践：写更可读的版本，看 EXPLAIN 决定。

**NOT IN vs NOT EXISTS**：当子查询有 NULL 时，`NOT IN` 永远返回 false（因为 NOT (x = NULL) 是 unknown）。`NOT EXISTS` 不受影响。生产上倾向 `NOT EXISTS` 避免坑。

### 5.9 ORDER BY 的底层原理（filesort 和 index）？

**答**：

**情况 1：用索引排序（最快）**

ORDER BY 字段恰好是索引顺序，直接按索引顺序读取，无需排序。EXPLAIN Extra 不显示 `Using filesort`。

**情况 2：filesort（额外排序）**

无法利用索引时，MySQL 把数据读到 sort buffer 排序。两种算法：

- **单路排序（modified file sort）**：把所有需要的字段读入内存排序。快但占内存。
- **双路排序（two-way file sort）**：只读 sort 字段 + 主键，排好后回表取字段。省内存但慢。

由 `max_length_for_sort_data` 决定：行长度小于阈值用单路，否则用双路。

排序数据量超过 `sort_buffer_size` 时用磁盘临时文件，性能急剧下降。

**优化思路**：

1. 让 ORDER BY 走索引（联合索引设计时考虑）
2. SELECT 只取需要的字段（减少行长度）
3. 增大 sort_buffer_size

### 5.10 GROUP BY 的优化思路？

**答**：

GROUP BY 默认会做排序（5.7 及以前）。优化：

1. **GROUP BY 字段加索引**：避免排序
2. **GROUP BY 与 ORDER BY 字段一致**：能利用同一个索引
3. **不需要排序时加 ORDER BY NULL**：5.7 及以前
4. **MySQL 8.0+ 不再隐式排序**：默认不排，行为更符合标准 SQL
5. **避免 GROUP BY 后跟函数**：`GROUP BY DATE(create_time)` 无法用索引

### 5.11 LIKE '%xxx%' 为什么索引失效？如何优化？

**答**：

B+ 树索引按字符前缀有序排列。`LIKE 'xxx%'` 能利用最左前缀走 range；`LIKE '%xxx%'` 没有确定的前缀，只能全索引扫描。

**优化方案**：

1. **业务上禁止前导通配符**：明确告诉用户只支持后缀模糊
2. **倒排索引**：把字符串反转存储一列，查询时也反转，`LIKE 'cba%'` 等价于查原表 `LIKE '%abc'`
3. **全文索引**：MySQL 5.7+ 支持中文 ngram 全文索引
4. **专业搜索引擎**：ES / Solr 处理模糊查询

## 六、高可用与架构

### 6.1 MySQL 主从复制的原理是什么？

**答**：

**三个线程**：

1. **Master Binlog Dump Thread**：监听 Slave 的请求，把 binlog 推给 Slave
2. **Slave IO Thread**：接收 binlog，写入本地 relay log（中继日志）
3. **Slave SQL Thread**：读 relay log，重放 SQL

**流程**：

1. Master 写 binlog
2. Slave IO Thread 连 Master，请求从某 binlog 位点开始的数据
3. Master Dump Thread 把对应数据发给 Slave
4. Slave IO Thread 写入 relay log
5. Slave SQL Thread 应用 relay log 中的事件

**binlog 格式**：

- **STATEMENT**：记录 SQL 语句。日志小，但某些 SQL 在主从可能产生不一致（如 NOW()、UUID()）
- **ROW**：记录每行的修改。安全但日志大
- **MIXED**：自动选择，安全时用 STATEMENT，不安全时用 ROW

生产推荐 ROW 格式。

### 6.2 主从延迟的原因及解决方案？

**答**：

**原因**：

1. **SQL Thread 单线程**（5.6 及以前）：Master 多线程并发写，Slave 单线程重放，跟不上
2. **大事务**：一个大事务占满 binlog dump 带宽
3. **网络延迟**
4. **从库性能差**（硬盘、CPU）
5. **从库负载高**：被业务读爆
6. **慢 SQL 阻塞**

**解决方案**：

1. **并行复制**：5.7+ 支持基于 LOGICAL_CLOCK 的多线程复制（同组事务可并行）
2. **拆分大事务**：限制每个事务影响的行数
3. **半同步复制**：保证至少一个从库收到 binlog 才提交
4. **从库升配置**：SSD、更多 CPU
5. **业务读主优化**：对延迟敏感的读直接走主库
6. **MySQL Group Replication / MGR**：多主同步复制

### 6.3 半同步复制和异步复制的区别？

**答**：

| | 异步复制 | 半同步复制 |
|:---|:---|:---|
| 流程 | Master 写完 binlog 立即返回 | Master 等至少一个 Slave 确认收到 binlog 才返回 |
| 性能 | 最快 | 慢（多一次网络往返）|
| 数据安全 | Master 挂了可能丢未同步的数据 | 至少一个 Slave 有数据 |
| 配置 | 默认 | 装 `rpl_semi_sync_master` 插件 |

半同步可降级为异步（超时 `rpl_semi_sync_master_timeout` 默认 10 秒）。

**全同步**：MGR / Galera Cluster 提供，性能更差但强一致。

### 6.4 读写分离的方案有哪些？

**答**：

**方案 1：应用层代码控制**

业务代码里手动选数据源：

```java
@Master 注解的方法走主库
@Slave 注解的方法走从库
```

优点：精细控制。缺点：侵入业务代码。

**方案 2：动态数据源（推荐）**

用 Spring AbstractRoutingDataSource，AOP 拦截方法决定路由。常见框架：ShardingSphere、Dynamic-Datasource。

**方案 3：中间件代理**

应用连代理，代理根据 SQL 类型路由。如 MyCat、ShardingSphere-Proxy、ProxySQL。

**方案 4：连接池层**

如 MySQL 官方的 MySQL Router，根据查询路由。

**注意**：

- 读写分离后必须考虑主从延迟（刚写完立即查可能查不到）
- 关键读操作仍要走主库
- 事务里所有读都走主库

### 6.5 分库分表的策略有哪些？水平分表和垂直分表的区别？

**答**：

**垂直分表**：按字段拆。常用 + 不常用字段分两张表。

```
user_basic(id, name, phone, ...)
user_extra(id, bio, avatar_url, preferences, ...)
```

**垂直分库**：按业务拆。订单库、用户库、商品库各自独立。

**水平分表**：按行拆。同一张逻辑表拆成多张物理表。

- **按范围**：order_2024_01、order_2024_02
- **按哈希**：order_0、order_1、order_2、order_3（按 id % 4 路由）
- **按地区**：order_beijing、order_shanghai

**水平分库**：分到多个 MySQL 实例。

**对比**：

- 垂直分：解决单表字段多 / 业务耦合
- 水平分：解决单表数据量大 / 单机性能瓶颈

通常先垂直再水平。

### 6.6 分库分表后如何处理跨库查询、分布式事务？

**答**：

**跨库查询**：

1. **冗余字段**：把常用的关联字段冗余到主表
2. **应用层组装**：在应用代码里多次查询合并
3. **数据同步**：用 binlog 同步成宽表到 ES / ClickHouse
4. **全局表**：少量数据的表（如字典）在每个库都复制一份

**分布式事务**：

1. **2PC（两阶段提交）**：XA 事务，性能差，强一致
2. **TCC**：Try-Confirm-Cancel，业务层补偿
3. **本地消息表**：业务表 + 消息表同库事务，异步发消息保证最终一致
4. **可靠消息（RocketMQ 事务消息）**：用 MQ 保证最终一致
5. **Seata**：开源分布式事务框架，支持 AT / TCC / XA 模式

互联网场景 90% 用最终一致，避免 XA。

### 6.7 什么是 XA 事务？

**答**：基于两阶段提交（2PC）的分布式事务规范。

**两阶段**：

1. **Prepare**：协调者通知所有参与者准备提交，参与者写好 redo log 但不提交，回复 yes/no
2. **Commit**：所有参与者都 yes 则统一提交，否则统一回滚

**MySQL XA 用法**：

```sql
XA START 'gtid';
INSERT INTO ...;
XA END 'gtid';
XA PREPARE 'gtid';
XA COMMIT 'gtid';
```

**问题**：

- 协调者单点：协调者挂了，参与者卡在 prepare 状态
- 同步阻塞：整个流程串行，性能差
- 部分失败：网络抖动可能导致部分提交部分未提交

所以互联网场景一般避免 XA，用最终一致代替。

### 6.8 ShardingSphere 和 MyCat 的区别？

**答**：

| | ShardingSphere | MyCat |
|:---|:---|:---|
| 形态 | Jar（嵌入应用）/ Proxy | Proxy |
| 维护 | Apache 顶级项目，活跃 | 社区维护，迭代慢 |
| 性能 | Jar 模式零开销 | 有代理跳数损耗 |
| 配置 | YAML / Spring 集成 | XML |
| 功能 | 分库分表 + 读写分离 + 影子库 + 数据加密 | 主要分库分表 |
| 推荐 | 新项目首选 | 老项目维护 |

ShardingSphere-JDBC 模式无网络开销，运维简单（每个应用自己分片）；ShardingSphere-Proxy 模式像 MyCat，集中化部署、支持多语言客户端。

## 附录：高频实战场景应对

### A.1 千万级数据 ALTER TABLE 怎么做？

**答**：直接 `ALTER TABLE` 会锁表很久。

1. **pt-online-schema-change**：Percona 工具，原理是创建新表 + 触发器同步 + 切换
2. **gh-ost**：GitHub 开源，基于 binlog 同步，比 pt-osc 对主库压力更小
3. **MySQL 8 Instant DDL**：部分 ALTER（加列、改默认值）秒级完成，无需重建表

### A.2 误删数据怎么恢复？

**答**：

1. **立即停止写入**（避免覆盖）
2. **看是否有备份**：拿最近备份恢复到误删前时间点
3. **binlog 回放**：用 `mysqlbinlog --start-datetime=... --stop-datetime=...` 提取误删前的 binlog，再应用
4. **专业工具**：用 `Xtrabackup` + binlog 做 point-in-time recovery
5. **如果有从库**：从库 stop slave，把数据导回来

预防：

- 关键操作前 `BEGIN; ... ;` 看到影响行数再 COMMIT
- 用 sql_safe_updates 防止 UPDATE / DELETE 不带 WHERE
- 重要库每小时 binlog 备份
- 定期演练恢复流程

### A.3 单表多大要考虑分库分表？

**答**：经验值：

- 数据量超过 1000 万行
- 单表存储超过 10GB
- 高峰 QPS 超过 1000

但不是绝对，要看：

- 是否有热点（部分行频繁访问）
- 是否有复杂 JOIN
- 是否需要全表扫描

很多时候先做这些优化就够了，不必上分库分表：

1. 加索引
2. 冷热分离（历史数据归档）
3. 读写分离
4. 加缓存

分库分表是最后的手段（复杂度大幅上升）。
