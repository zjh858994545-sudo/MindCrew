# 企业文档：版本发布流程与上线检查

## 文档目标

本文档用于定义企业系统版本发布的推荐流程，涵盖需求冻结、测试验收、灰度上线、观察期和回滚条件。

## 一、发布目标

一个成熟的发布流程应同时满足：

- 上线可控
- 风险可回退
- 责任明确
- 信息同步

## 二、角色分工

### 产品负责人

- 确认需求范围
- 明确是否允许变更延期
- 确认发布公告内容

### 研发负责人

- 确认代码合并范围
- 确认部署包版本
- 提供回滚方案

### 测试负责人

- 确认回归结果
- 给出是否可发布结论

### 运维或平台负责人

- 执行部署
- 观察上线指标
- 触发回滚操作

## 三、标准发布流程

### 1. 冻结发布范围

在发布前应明确：

- 哪些功能进入本次发布
- 哪些缺陷必须修复
- 哪些需求延后

### 2. 通过测试验收

至少包括：

- 核心功能回归
- 关键接口验证
- 问答主链验证
- 来源展示验证
- 关键配置验证

### 3. 准备发布包和变更单

发布前应准备：

- 构建版本号
- 变更说明
- 配置变更项
- 数据脚本
- 回滚步骤

### 4. 执行上线

推荐顺序：

1. 备份关键数据
2. 执行配置变更
3. 发布应用
4. 做健康检查
5. 做冒烟验证

### 5. 观察期

发布后应重点观察：

- 错误率
- 响应时间
- 核心页面与接口
- 问答成功率
- 工具调用成功率

## 四、灰度发布建议

适用于：

- 变更范围大
- 风险未知
- 涉及核心链路

常见方式：

- 指定用户灰度
- 指定流量比例灰度
- 指定租户灰度

灰度阶段重点确认：

- 新功能是否符合预期
- 老功能是否出现回归
- 指标是否稳定

## 五、回滚触发条件

建议提前定义清晰的回滚条件，例如：

- 核心接口连续异常
- 登录或鉴权失败
- 问答主链不可用
- 严重数据污染风险
- 用户投诉集中爆发

## 六、上线检查清单

### 上线前

- 版本号正确
- 配置文件正确
- 数据库脚本已审核
- 回滚方案已确认
- 责任人在线

### 上线后

- 首页可访问
- 登录正常
- 知识库查询正常
- SSE 问答正常
- MCP 工具可见

## 七、发布说明建议内容

每次发布说明至少包含：

- 版本号
- 发布时间
- 变更摘要
- 风险提示
- 升级说明
- 回滚说明
- 已知限制

## 八、总结

一个好发布流程的关键不是“步骤很多”，而是“每一步都能降低不确定性”。对知识库问答系统而言，发布时尤其要关注检索链路、来源展示、配置正确性和工具可用性。
