# 八股文 · 网络

> 覆盖 HTTP、TCP/UDP、网络基础、安全四大主题。

## 一、HTTP

### 1.1 HTTP 1.0、1.1、2.0、3.0 的区别

**答**：

| | HTTP 1.0 | HTTP 1.1 | HTTP 2.0 | HTTP 3.0 |
|:---|:---|:---|:---|:---|
| 传输协议 | TCP | TCP | TCP | QUIC（UDP）|
| 连接 | 短连接（每次新建）| 长连接（keep-alive）| 多路复用 | 多路复用（无队头阻塞）|
| 数据格式 | 文本 | 文本 | 二进制帧 | 二进制帧 |
| 请求并行 | 串行 | 管道化（基本不用）| 完全并行 | 完全并行 |
| 头压缩 | 无 | 无 | HPACK | QPACK |
| 服务器推送 | 无 | 无 | 支持 | 支持 |
| 队头阻塞 | 严重 | 严重 | TCP 层仍有 | 完全消除 |

**1.0**：每次请求建一个 TCP 连接，请求结束就断。**短连接性能差**。

**1.1（1997，至今主流）**：

- 默认 Keep-Alive 长连接
- 管道化（pipelining，但实际很少用）
- 引入 Host 头（支持虚拟主机）
- 分块传输编码（chunked）

**2.0（2015）**：

- 多路复用：一个 TCP 连接同时跑多个请求
- 二进制分帧
- HPACK 头压缩
- 服务器推送（Server Push）
- **问题**：基于 TCP，TCP 层丢包仍会队头阻塞

**3.0（2022 标准化）**：

- 基于 QUIC（UDP + TLS 1.3）
- 完全消除队头阻塞
- 0-RTT / 1-RTT 握手（之前 3-RTT）
- 连接迁移（IP 切换不断连接，移动端友好）

### 1.2 HTTP 和 HTTPS 的区别

**答**：

| | HTTP | HTTPS |
|:---|:---|:---|
| 端口 | 80 | 443 |
| 协议 | 明文 | TLS 加密 |
| 性能 | 快 | 略慢（握手 + 加密开销）|
| CA 证书 | 不需要 | 需要（信任链）|
| 安全 | 数据可被中间人窃取 / 篡改 | 加密 + 完整性 + 身份验证 |

**HTTPS = HTTP + TLS**。在 HTTP 和 TCP 之间加一层 TLS 做加密和身份认证。

**HTTPS 性能损耗**：

- 握手开销（首次 1-2 个 RTT）
- 加密 / 解密 CPU 开销（现代 CPU 有 AES-NI 指令集，影响很小）
- TLS 1.3 + HTTP/2 后基本可以忽略

### 1.3 HTTPS 的握手过程是什么？TLS 1.2 和 1.3 的区别？

**答**：

**TLS 1.2 握手（2-RTT）**：

```
Client                                Server
   |  --- ClientHello ---->             |  ← 支持的加密套件 + 随机数A
   |  <-- ServerHello -----             |  ← 选定的加密套件 + 随机数B
   |  <-- Certificate -----             |  ← 服务器证书
   |  <-- ServerHelloDone -             |
   |  --- ClientKeyExchange→            |  ← 用证书公钥加密随机数C（pre-master）
   |  --- ChangeCipherSpec→             |
   |  --- Finished ------->             |
   |  <-- ChangeCipherSpec-             |
   |  <-- Finished -------              |
   |  ===== 应用数据加密传输 =====       |
```

最终双方都有 A/B/C 三个随机数，由此推导出**会话密钥**（对称加密）。

**TLS 1.3 优化（1-RTT，重连 0-RTT）**：

- 简化握手流程，ClientHello 时就发送密钥参数
- 减少加密套件（移除不安全的）
- 强制前向安全（每次会话都有独立密钥）
- 0-RTT 模式：再次连接同一服务器时直接发数据，没有任何握手延迟

**两者区别**：

| | TLS 1.2 | TLS 1.3 |
|:---|:---|:---|
| 握手 RTT | 2 | 1 / 0（恢复）|
| 加密套件 | 多（含弱算法）| 精简（强算法）|
| 前向安全 | 可选 | 强制 |
| 0-RTT | 不支持 | 支持 |

**SSL/TLS 用对称加密还是非对称**？

- 握手阶段用**非对称**（RSA / ECDHE）安全交换密钥
- 数据传输用**对称**（AES）保证性能

### 1.4 HTTP 的常见状态码？301 和 302 的区别？304 是什么？

**答**：

**五大类**：

- 1xx 信息（很少用）：100 Continue
- 2xx 成功：
  - 200 OK
  - 201 Created
  - 204 No Content
- 3xx 重定向：
  - 301 永久重定向
  - 302 临时重定向
  - 304 Not Modified（协商缓存命中）
  - 307 临时重定向（不允许改 method）
  - 308 永久重定向（不允许改 method）
- 4xx 客户端错误：
  - 400 Bad Request
  - 401 Unauthorized
  - 403 Forbidden
  - 404 Not Found
  - 405 Method Not Allowed
  - 408 Request Timeout
  - 429 Too Many Requests
- 5xx 服务端错误：
  - 500 Internal Server Error
  - 502 Bad Gateway
  - 503 Service Unavailable
  - 504 Gateway Timeout

**301 vs 302**：

| | 301 | 302 |
|:---|:---|:---|
| 含义 | 永久重定向 | 临时重定向 |
| 缓存 | 浏览器会缓存 | 不缓存 |
| SEO | 旧 URL 的权重转移到新 URL | 不转移 |
| 用途 | 网站迁移 / HTTPS 强制 | 临时跳转登录页 |

**304 Not Modified**：协商缓存命中。浏览器带 `If-Modified-Since` / `If-None-Match` 请求，服务端判断资源没变，返回 304 + 空 body，浏览器用本地缓存。

### 1.5 GET 和 POST 的区别

**答**：

| | GET | POST |
|:---|:---|:---|
| 语义 | 查询 | 提交 |
| 参数位置 | URL（querystring） | body |
| 长度限制 | 浏览器/服务器限制 URL 长度（约 2KB-8KB）| 理论无限（实际看配置）|
| 缓存 | 可缓存 | 不缓存（除非显式声明）|
| 浏览历史 | 保留 | 不保留 |
| 收藏 | 可收藏（含参数）| 不可 |
| 安全 | 参数在 URL 中可见 | body 不在 URL，但 HTTP 同样明文 |
| 幂等 | 是 | 否 |

**澄清几个误区**：

- "POST 比 GET 安全"——错。两者都是明文，要安全用 HTTPS。POST 参数不在 URL 不代表不可见
- "GET 不能传 body"——HTTP 协议允许，但很多工具忽略，实际不推荐
- "POST 发 2 个 TCP 包"——只是某些 curl/浏览器实现细节，不是协议规定

### 1.6 HTTP 的 keep-alive 是什么？

**答**：

**TCP 层 KeepAlive**：检测连接是否还活着。空闲一段时间发探测包，没回响就断开。跟 HTTP 无关。

**HTTP 层 Keep-Alive**（连接复用）：在一个 TCP 连接上发送多个 HTTP 请求 / 响应。

```
HTTP 1.0：默认关闭，要显式声明 Connection: Keep-Alive
HTTP 1.1：默认开启，要关闭显式声明 Connection: close
```

**好处**：

- 省去 TCP 三次握手 / 四次挥手
- 避免 TLS 重复握手
- 减少慢启动影响

**问题**：

- 队头阻塞：一个请求慢了卡住后面所有请求
- 连接资源占用：服务端要维护连接

**超时参数**：

- 服务端 `keepalive_timeout`：空闲多久关闭（Nginx 默认 65 秒）
- 客户端可以主动关闭

### 1.7 HTTP 的缓存机制（强缓存和协商缓存）

**答**：

**两级缓存**：

**1. 强缓存**：浏览器不发请求，直接用本地缓存。

控制 header：

- **Cache-Control**（HTTP 1.1，优先）：
  - `max-age=3600`：缓存 1 小时
  - `no-cache`：每次都协商缓存
  - `no-store`：禁止任何缓存
  - `public` / `private`：是否允许中间代理缓存
- **Expires**（HTTP 1.0，绝对时间）：`Expires: Wed, 21 Oct 2024 07:28:00 GMT`

强缓存命中时返回 200 (from disk cache)。

**2. 协商缓存**：浏览器发请求带验证字段，服务器判断资源没变就返回 304。

控制 header：

- **Last-Modified** + **If-Modified-Since**：基于修改时间
- **ETag** + **If-None-Match**：基于内容哈希（精度更高）

```
首次请求响应：
Last-Modified: Wed, 21 Oct 2024 07:28:00 GMT
ETag: "abc123"

再次请求：
If-Modified-Since: Wed, 21 Oct 2024 07:28:00 GMT
If-None-Match: "abc123"

服务器判断未变 → 304
判断已变 → 200 + 新内容
```

**优先级**：

```
Cache-Control > Expires
ETag > Last-Modified
强缓存 → 协商缓存 → 真请求
```

**实战配置**：

- 静态资源（带版本号）：`Cache-Control: max-age=31536000, immutable`（长期强缓存）
- HTML：`Cache-Control: no-cache`（强制协商）
- API：`Cache-Control: no-store`（不缓存）

### 1.8 Cookie 和 Session 的区别

**答**：

| | Cookie | Session |
|:---|:---|:---|
| 存储位置 | 浏览器（客户端）| 服务器内存 / 数据库 |
| 安全性 | 用户可改 | 服务端可控 |
| 大小 | 单个 < 4KB | 受服务端容量限制 |
| 跨域 | 默认不发送（看 SameSite）| 不涉及 |
| 关联 | 通过 Cookie 中的 session_id 找到 Session | - |

**典型流程**：

```
用户登录 → 服务端创建 Session（含用户信息）→ 返回 Set-Cookie: session_id=abc
浏览器后续请求自动带 Cookie: session_id=abc
服务端根据 session_id 找到 Session → 知道用户身份
```

**Cookie 属性**：

- `HttpOnly`：JS 不能读（防 XSS）
- `Secure`：只能 HTTPS 传输
- `SameSite`：跨站发送策略（None / Lax / Strict）
- `Path` / `Domain`：作用范围
- `Max-Age` / `Expires`：过期时间

**SameSite**：

- `None`：跨站也发送（必须 Secure）
- `Lax`（默认）：导航跳转时发送（如点击外链），非导航 POST 不发
- `Strict`：完全不跨站发送

**JWT 跟 Session 的对比**：JWT 是 stateless 的，服务端不存。Session 是 stateful 的，服务端存。前者扩展性好但难撤销，后者反之。

### 1.9 HTTP 2.0 的多路复用和 HTTP 1.1 的 keep-alive 的区别？

**答**：

**HTTP 1.1 Keep-Alive**：

- 一个 TCP 连接可以**串行**发多个请求
- 同一时间只能处理一个请求（必须等响应完才能发下一个）
- 浏览器为了并发会开多个 TCP 连接（一般 6 个）

**HTTP 2.0 多路复用**：

- 一个 TCP 连接可以**并行**发多个请求 / 响应
- 通过"流（Stream）"区分不同请求
- 二进制分帧后可以乱序传输，接收端重组
- 浏览器只需要一个 TCP 连接

**对比**：

```
HTTP 1.1 Keep-Alive：
请求1 → 响应1 → 请求2 → 响应2 → 请求3 → 响应3
（串行）

HTTP 2.0 多路复用：
请求1 ────┐
请求2 ────┼──→ Server
请求3 ────┘
       ↓ 同时返回
响应3 ────┐
响应1 ────┤
响应2 ────┘
（并行 + 可乱序）
```

**问题**：HTTP 2.0 在 TCP 层仍有队头阻塞——TCP 丢包重传时，整个连接所有请求都等。HTTP 3.0 用 QUIC 解决。

### 1.10 HTTP 3.0 基于 QUIC 的原理

**答**：

**QUIC = Quick UDP Internet Connections**，Google 提出。

**核心特点**：

**1. 基于 UDP**：

- UDP 简单，没 TCP 那些握手 / 拥塞控制约束
- 在 UDP 上自己实现可靠性 + 流量控制 + 加密

**2. 内置 TLS 1.3**：

- 不是分层的（TCP + TLS）而是融合的
- 握手减少到 1-RTT，重连 0-RTT

**3. 多路流，无队头阻塞**：

- 同一连接多个流互不影响
- 单流丢包重传只影响该流，其他流继续

**4. 连接迁移**：

- 用 Connection ID 而不是 IP+Port 标识连接
- 手机从 WiFi 切到 4G，IP 变了但连接不断
- 移动端用户体验大幅提升

**5. 改进的拥塞控制**：

- 用户态实现，更灵活
- 支持 BBR 等现代算法

**部署现状**：

- Chrome / Edge / Firefox 已支持
- Google / Facebook / 阿里云 / 腾讯云 等头部站点已上 HTTP/3
- CDN 广泛支持
- 企业自建服务推广较慢（防火墙对 UDP 不友好）

## 二、TCP/UDP

### 2.1 TCP 和 UDP 的区别及各自适用场景？

**答**：

| | TCP | UDP |
|:---|:---|:---|
| 连接 | 面向连接（三次握手）| 无连接 |
| 可靠性 | 可靠（ACK + 重传）| 不保证 |
| 顺序 | 保证 | 不保证 |
| 流量控制 | 滑动窗口 | 无 |
| 拥塞控制 | 有 | 无 |
| 头部开销 | 20 字节 | 8 字节 |
| 传输 | 字节流 | 数据报 |
| 性能 | 较慢 | 快 |
| 适用场景 | HTTP/FTP/SMTP/SSH | DNS/视频/直播/游戏/RTC |

**TCP 适用**：

- 浏览器请求
- 文件传输（FTP）
- 邮件（SMTP/POP3/IMAP）
- 数据库连接

**UDP 适用**：

- DNS 查询（小包 + 速度）
- 视频流 / 直播（丢一帧不重要）
- 游戏（实时性 > 完整性）
- 音视频会议
- IoT 设备
- QUIC / HTTP 3.0

### 2.2 TCP 三次握手的过程？为什么是三次而不是两次？

**答**：

```
Client                          Server
  |                                |
  |  --- SYN(seq=x) --->           |   ① 客户端发起，状态变 SYN_SENT
  |                                |
  |  <-- SYN(seq=y) ACK(x+1) ---   |   ② 服务端回应，状态变 SYN_RCVD
  |                                |
  |  --- ACK(y+1) --->             |   ③ 客户端确认
  |                                |
  | ====== ESTABLISHED ======      |
```

**为什么三次不是两次**？

如果两次：客户端发 SYN → 服务端发 SYN+ACK → 直接连接。

**问题**：

- 客户端 SYN 在网络中迟到，服务端收到后建连，但客户端已经放弃了
- 服务端建好的连接没人用，浪费资源
- 客户端的旧请求可能造成数据混乱

第三次握手让客户端确认"我收到你的回复了，确实要连"，避免上述问题。

**简洁解释**：

- 第一次：服务端确认能收到客户端的消息
- 第二次：客户端确认能收到服务端的消息（双向 OK）
- 第三次：服务端确认客户端收到了自己的回复

### 2.3 TCP 四次挥手的过程？为什么需要 TIME_WAIT？

**答**：

```
Client                          Server
  |                                |
  |  --- FIN(seq=x) --->           |  ① 客户端断开，进 FIN_WAIT_1
  |                                |
  |  <-- ACK(x+1) ----             |  ② 服务端 ACK，进 CLOSE_WAIT
  |                                |     客户端进 FIN_WAIT_2
  |                                |
  |  <-- FIN(seq=y) ----           |  ③ 服务端处理完后发 FIN，进 LAST_ACK
  |                                |
  |  --- ACK(y+1) --->             |  ④ 客户端 ACK，进 TIME_WAIT
  |                                |
  | === 2MSL 后 CLOSED ===         |     服务端收到 ACK 后 CLOSED
```

**为什么四次而不是三次**？

服务端的 ACK 和 FIN 不能合并（不像三次握手时服务端的 SYN 和 ACK 可以一起发），因为服务端收到客户端 FIN 时可能还有数据没发完，需要先 ACK 让客户端知道收到了，等数据发完再发 FIN。

**为什么 TIME_WAIT 需要 2MSL**？

**MSL（Maximum Segment Lifetime）** = 报文最大生存时间（一般 30 秒-2 分钟）。

两个原因：

1. **确保最后 ACK 能到对方**：如果 ACK 丢了，服务端会重发 FIN。客户端在 TIME_WAIT 期间还能响应。如果立即 CLOSED，重发的 FIN 没人理，服务端会一直等。
2. **防止旧连接的报文影响新连接**：等 2MSL 让网络中残余报文消失，避免下一个相同 5 元组的连接收到老报文。

### 2.4 TCP 的可靠传输是如何保证的？

**答**：

**1. 序号 + 确认**

每个字节有序号，接收方对每个收到的字节段回 ACK，告诉发送方"我收到 X 之前的了，下一个等你发 X+1"。

**2. 超时重传**

发送方发完数据启动定时器，没收到 ACK 就重发。RTO（重传超时）根据 RTT 动态调整。

**3. 快速重传**

接收方收到乱序段时立即重复 ACK 上一次确认的序号。发送方收到 3 个重复 ACK 立即重发该段（不等超时）。

**4. 流量控制**

接收方在 ACK 中告知自己的接收窗口大小（rwnd），发送方不会发超过 rwnd 的数据，避免压垮接收方。

**5. 拥塞控制**

四个算法（慢启动、拥塞避免、快速重传、快速恢复）控制发送速度，避免拥塞整个网络。

**6. 数据校验**

TCP 头部有 checksum，接收方校验失败的段会被丢弃 → 发送方超时重传。

### 2.5 TCP 的流量控制和拥塞控制的区别？

**答**：

| | 流量控制 | 拥塞控制 |
|:---|:---|:---|
| 目的 | 防止压垮**接收方** | 防止拥塞**网络** |
| 控制方 | 接收方告知发送方 | 发送方自己判断 |
| 实现 | rwnd（接收窗口） | cwnd（拥塞窗口） |
| 实际发送量 | min(rwnd, cwnd) | |

**流量控制**：接收方在 ACK 里带 `Window` 字段告诉发送方"我现在能收多少"。如果接收方处理慢，rwnd 减小，发送方放慢。极端情况 rwnd=0，发送方停发（但定期发 0 窗口探测包问能不能继续）。

**拥塞控制**四算法：

**1. 慢启动**：cwnd 从 1 开始，每收到一个 ACK，cwnd +1。看似线性实际指数（每个 RTT 翻倍）。

**2. 拥塞避免**：cwnd 达到 ssthresh（阈值）后，每个 RTT cwnd 加 1（线性增长）。

**3. 快速重传**：收到 3 个重复 ACK 立即重发，不等超时。

**4. 快速恢复**：进入快重传后，ssthresh 设为当前 cwnd 的一半，cwnd = ssthresh + 3，跳过慢启动直接进入拥塞避免。

发生超时：ssthresh = cwnd/2，cwnd = 1，重新慢启动。

### 2.6 TCP 的滑动窗口机制是什么？

**答**：

**目的**：实现流量控制 + 提高吞吐。

**核心思想**：发送方不必每发一个段就等 ACK，可以批量发"窗口内"的数据，提高效率。

```
已发送已确认 | 已发送未确认 | 可发送未发送 | 不可发送
            ↑              ↑
         窗口左边界       窗口右边界（接收方控制）
```

**窗口滑动**：收到 ACK 后窗口右移，腾出空间发新数据。

**接收方窗口**：

```
已确认 | 未确认（窗口）| 未到达
```

接收方在 ACK 里带剩余窗口大小，发送方根据这个调整发送量。

**优化场景**：

- **糊涂窗口**：接收方处理慢，每次只腾一点点空间，发送方就发一点点，效率低。解决：等窗口达到一定大小再通告。
- **Nagle 算法**：发送方累积小包合并发送（最多 1 个未确认的小包在网络中）。可关闭（`TCP_NODELAY`）以降低延迟。

### 2.7 TCP 的粘包和拆包问题是什么？如何解决？

**答**：

TCP 是**面向字节流**的，数据没有"消息边界"。两次 send 可能被 OS 合并发送（粘包），一次 send 也可能被拆成多次到达（拆包）。

**示例**：

```
Send "Hello" + Send "World"
Receive 可能：
  "HelloWorld"     ← 粘包
  "He" "lloWorld"  ← 拆包
  "HelloWor" "ld"  ← 半粘半拆
```

**注意**：UDP 没这问题（面向报文，每次接收一个完整数据报）。

**解决方案**：

**1. 固定长度**

每个消息固定 N 字节。简单但不灵活，浪费空间。

**2. 分隔符**

消息之间加特殊字符（如 `\n` 或 `\r\n`）。HTTP 头就是这种。问题：消息内容不能含分隔符（要转义）。

**3. 长度字段（最常用）**

消息前面加固定字节数表示长度：

```
[4 bytes length] [body]
```

读取时：先读 4 字节得长度 N → 再读 N 字节就是完整消息。

**4. 协议格式**

定义完整的协议格式（如 Protobuf、Thrift）自动处理边界。

**框架支持**：

- Netty：`LengthFieldBasedFrameDecoder` 自动按长度字段拆包
- gRPC：基于 HTTP/2 帧自动处理

### 2.8 SYN flood 攻击是什么？如何防范？

**答**：

**攻击原理**：

攻击者发大量 SYN 包但不回最后的 ACK。服务端为每个 SYN 分配资源进入 SYN_RCVD 状态（半连接队列），队列满了正常用户连不上。

**防范**：

**1. SYN Cookies**

服务端不立即分配资源。把连接信息编码到 SYN+ACK 的初始序号中，等收到 ACK 时再用序号验证。无状态防御。

```bash
sysctl -w net.ipv4.tcp_syncookies=1
```

**2. 加大半连接队列**

```bash
sysctl -w net.ipv4.tcp_max_syn_backlog=8192
```

**3. 减少 SYN+ACK 重试次数**

默认 5 次重试约 63 秒。降到 2-3 次缩短资源占用。

```bash
sysctl -w net.ipv4.tcp_synack_retries=2
```

**4. 防火墙 / WAF / 流量清洗**

云厂商的 DDoS 防护服务（阿里云盾、腾讯云大禹）。

### 2.9 TIME_WAIT 过多如何解决？

**答**：

**TIME_WAIT 出现在主动关闭方**。如果服务器主动关闭大量短连接，TIME_WAIT 会堆积。

**问题**：

- 占用端口（每个 TIME_WAIT 占一个本地端口）
- 占内存
- 端口耗尽（默认本地端口范围有限）

**解决**：

**1. 修改长连接策略**

- HTTP Keep-Alive
- 数据库连接池
- gRPC / Dubbo 默认长连接

**2. 端口复用**

```bash
sysctl -w net.ipv4.tcp_tw_reuse=1
```

允许 TIME_WAIT 状态的端口被新连接复用（仅作为客户端时）。

**3. 减小 TIME_WAIT 时长**（不推荐）

```bash
sysctl -w net.ipv4.tcp_fin_timeout=30
```

**4. 加大端口范围**

```bash
sysctl -w net.ipv4.ip_local_port_range="1024 65535"
```

**5. 业务层调整**

让客户端主动关闭连接而不是服务端。

注意：`net.ipv4.tcp_tw_recycle` 在 Linux 4.12 已移除（NAT 环境下有问题），不要用。

### 2.10 CLOSE_WAIT 过多是什么原因？

**答**：

**CLOSE_WAIT 出现在被动关闭方**。对端发了 FIN，本端 ACK 后进入 CLOSE_WAIT，等待应用层调用 `close()` 发送 FIN。

**CLOSE_WAIT 堆积 = 应用层没有正确关闭连接**。

**常见原因**：

1. **代码 bug**：异常分支没 `close()` / `try-finally` 缺失
2. **连接池泄漏**：拿连接后没归还
3. **依赖服务慢**：业务逻辑卡住，没走到 `close()`
4. **socket close 顺序问题**：先 close 输入流后 close socket 等

**排查**：

```bash
netstat -ant | grep CLOSE_WAIT | awk '{print $5}' | sort | uniq -c
# 看哪个对端 CLOSE_WAIT 多

ss -tnlp | grep CLOSE_WAIT
# 看是哪个进程
```

**修复**：

- 用 `try-with-resources`（Java）或 `defer`（Go）保证关闭
- 配连接池超时 + 最大空闲时间
- 监控应用线程 stack 看是否卡住

## 三、网络基础

### 3.1 OSI 七层模型和 TCP/IP 四层模型的对应关系

**答**：

| OSI 七层 | TCP/IP 四层 | 协议示例 | 数据单位 |
|:---|:---|:---|:---|
| 应用层 | 应用层 | HTTP/FTP/DNS/SMTP | 报文 |
| 表示层 | 应用层 | TLS/SSL | - |
| 会话层 | 应用层 | - | - |
| 传输层 | 传输层 | TCP/UDP | 段（Segment）|
| 网络层 | 网络层 | IP/ICMP/ARP | 包（Packet）|
| 数据链路层 | 网络接入层 | 以太网/PPP | 帧（Frame）|
| 物理层 | 网络接入层 | 电信号 / 光纤 | 比特 |

**OSI** 是理论模型，**TCP/IP** 是工程实现。

**简记**：

- 应用层：处理什么数据（HTTP 报文 / DNS 查询 / 邮件）
- 传输层：怎么传（可靠 TCP / 不可靠 UDP）
- 网络层：传到哪（IP 路由）
- 链路层：怎么走（MAC 地址 / 网卡）

### 3.2 DNS 解析的完整过程

**答**：

```
浏览器输入 example.com
  ↓
1. 浏览器 DNS 缓存
  ↓ miss
2. 操作系统 DNS 缓存（hosts 文件 + 系统缓存）
  ↓ miss
3. 本地 DNS 服务器（运营商 / 路由器，递归查询）
  ↓ miss
4. 本地 DNS 向 Root DNS 查询
  ↓ Root 返回 .com 服务器地址
5. 本地 DNS 向 .com 服务器查询
  ↓ .com 返回 example.com 的权威 NS 服务器
6. 本地 DNS 向权威 NS 查询
  ↓ 返回 IP 地址
7. 本地 DNS 返回给操作系统 → 返回给浏览器
8. 浏览器用 IP 建立连接
```

**两种查询方式**：

- **递归查询**：客户端 → 本地 DNS（要么给答案要么给错误）
- **迭代查询**：本地 DNS → 根 → 顶级域 → 权威（一步步问）

**优化**：

- DNS 缓存（多级）
- HTTPDNS（绕过本地 DNS，避免劫持）
- DNS over HTTPS / TLS（加密 DNS）
- CDN 智能解析（按地理位置返回不同 IP）

### 3.3 ARP 协议是什么？

**答**：

**ARP（Address Resolution Protocol）**：IP 地址 → MAC 地址的解析。工作在链路层和网络层之间。

**为什么需要**：

- 网络层用 IP 寻址
- 链路层用 MAC 寻址
- 同一局域网内通信，需要把 IP 转成 MAC

**工作流程**（A 要给同网段的 B 发包）：

1. A 查自己的 ARP 缓存，没有 B 的 MAC
2. A 广播 ARP 请求："谁是 192.168.1.5？告诉我"
3. 同网段所有主机收到，只有 B 应答："我是 192.168.1.5，我的 MAC 是 xx:xx:xx"
4. A 收到回复，缓存到本地 ARP 表
5. A 用 B 的 MAC 发数据

**跨网段**：A 把包发给默认网关，网关代为转发（ARP 解析的是网关的 MAC，不是 B 的）。

**ARP 欺骗**：攻击者伪造 ARP 应答，让目标把流量发给自己。在同网段是真实威胁。

### 3.4 浏览器输入 URL 到页面显示的完整过程

**答**：

经典面试题，可以讲很深或很浅，按需展开。

**简版**：

1. **URL 解析**：协议、域名、端口、路径
2. **DNS 解析**：域名 → IP
3. **TCP 连接**：三次握手
4. **TLS 握手**（HTTPS）
5. **发送 HTTP 请求**
6. **服务端处理**：业务逻辑、查 DB、生成响应
7. **接收响应**
8. **断开 TCP 连接**（或 keep-alive 复用）
9. **浏览器解析**：HTML → DOM 树、CSS → CSSOM、合并成渲染树
10. **布局 + 绘制**：计算位置、绘制像素
11. **加载子资源**：JS / CSS / 图片
12. **JS 执行**：可能修改 DOM 触发重绘 / 重排

**详细版亮点**：

- 浏览器缓存 / DNS 缓存
- HTTP/2 多路复用、HTTP/3 QUIC
- Critical Rendering Path
- 浏览器主线程 / 渲染线程 / JS 线程
- 重绘 vs 重排

后端面试到第 6 步即可，前端要详细到 9-12 步。

### 3.5 CDN 的原理是什么？

**答**：

**CDN（Content Delivery Network）**：把静态内容（图片、视频、JS、CSS）放在离用户最近的服务器上，加速访问。

**核心原理**：

1. **就近访问**：用户请求时 DNS 智能解析到最近的 CDN 节点
2. **多级缓存**：边缘节点 → 区域节点 → 源站
3. **回源**：CDN 没有时回源站取，缓存后下次直接返回

**工作流程**：

```
用户访问 cdn.example.com/logo.png
  ↓
DNS 解析（CDN 服务商的智能 DNS 根据用户位置返回最近边缘节点 IP）
  ↓
请求边缘节点
  ↓
节点缓存命中 → 直接返回
节点未命中 → 回源 → 缓存 → 返回
```

**好处**：

- 降低延迟（地理就近）
- 减轻源站压力
- 提高可用性（节点宕机自动切换）
- DDoS 防护（分散流量）

**适用场景**：图片、视频、静态文件、API 缓存。

### 3.6 WebSocket 和 HTTP 的区别？

**答**：

| | HTTP | WebSocket |
|:---|:---|:---|
| 通信方向 | 客户端发起，单向请求 / 响应 | 双向 |
| 连接 | 短连接（Keep-Alive 也是逻辑长）| 长连接 |
| 协议 | 文本（HTTP）| 帧（自定义）|
| 适用 | 请求 / 响应模式 | 实时双向通信 |
| 数据 | 文本 + 二进制 | 文本 + 二进制 |
| 头部 | 每次都有 | 握手后没有 |

**WebSocket 握手**：

```
Client: 
GET /chat HTTP/1.1
Host: server.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==

Server:
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

← 之后用 WebSocket 协议双向通信
```

通过 HTTP 握手升级到 WebSocket 协议。

**适用场景**：

- 聊天
- 在线游戏
- 实时通知（订单状态变更推送）
- 协同编辑
- 股票行情

**对比 SSE**：SSE 是 HTTP 之上的单向推送，WebSocket 是双向。SSE 更轻量、防火墙友好；WebSocket 双向但实现复杂。

### 3.7 正向代理和反向代理的区别？

**答**：

**正向代理**：代理**客户端**。客户端无法直接访问服务器，通过代理转发。

```
客户端 → 正向代理 → 服务器
（服务器看到的是代理的 IP）
```

**例子**：VPN、浏览器 SOCKS 代理。客户端主动配置，知道自己在用代理。

**反向代理**：代理**服务器**。客户端以为自己直接访问服务器，实际访问的是代理，代理再转发到内部多台服务器。

```
客户端 → 反向代理 → 服务器集群
（客户端看到的是代理的 IP）
```

**例子**：Nginx、负载均衡器。客户端无感知，代理对外是单一入口。

**核心区别**：

| | 正向代理 | 反向代理 |
|:---|:---|:---|
| 代理对象 | 客户端 | 服务器 |
| 谁知道有代理 | 客户端 | 服务器（或两边都知道）|
| 隐藏 | 隐藏客户端 IP | 隐藏服务器结构 |
| 配置方 | 客户端主动配置 | 服务端部署 |
| 用途 | 翻墙 / 隐私 / 缓存 | 负载均衡 / 安全 / 缓存 |

### 3.8 长连接和短连接的区别及使用场景？

**答**：

**短连接**：每次通信建一个 TCP 连接，通信完就断。

**长连接**：连接建立后保持，多次通信复用。

| | 短连接 | 长连接 |
|:---|:---|:---|
| 建连开销 | 高（每次握手 + 挥手）| 低 |
| 资源占用 | 低 | 高（保持连接）|
| 适用 | 偶尔通信、连接数少 | 频繁通信、实时 |

**应用层选择**：

- HTTP 1.1+ 默认长连接（Keep-Alive）
- 数据库连接池：长连接
- 消息推送（IM）：长连接
- 内部 RPC（Dubbo / gRPC）：长连接

**保活机制**：长连接需要心跳防止中间设备（NAT、防火墙）超时关闭。一般 30-60 秒一次。

## 四、安全

### 4.1 XSS 和 CSRF 攻击的原理及防范

**答**：

**XSS（Cross-Site Scripting）跨站脚本攻击**：

攻击者在网页中注入恶意 JS，浏览器执行后窃取用户数据。

**三种类型**：

- **存储型**：恶意脚本存到数据库（如评论区注入 `<script>`），每个查看者都受害
- **反射型**：恶意脚本在 URL 参数里，服务端拼到响应里反射回去
- **DOM 型**：纯前端漏洞，JS 直接把 URL 参数注入 DOM

**防范**：

1. **输出转义**：把 `<`、`>`、`"`、`'`、`&` 转成 HTML entity
2. **CSP（Content Security Policy）**：`Content-Security-Policy: script-src 'self'` 只允许同源脚本
3. **HttpOnly Cookie**：让 JS 读不到 session cookie
4. **输入校验 / 白名单**：富文本编辑器用白名单标签
5. **使用安全的框架**：React / Vue 默认转义

**CSRF（Cross-Site Request Forgery）跨站请求伪造**：

利用用户已登录身份，诱骗其访问恶意网站，恶意网站向目标网站发请求（带上用户 cookie）。

**例子**：用户登录了银行 → 访问恶意网站 → 网站偷偷 POST 到银行转账接口（浏览器自动带 cookie）→ 转账成功。

**防范**：

1. **CSRF Token**：表单提交时带一个服务端生成的随机 token，攻击者无法跨域读取
2. **SameSite Cookie**：`Set-Cookie: ...; SameSite=Lax`，跨站不发 cookie
3. **检查 Referer / Origin**：拒绝非本站来源
4. **重要操作要二次确认**：密码、短信验证码

**XSS vs CSRF**：

- XSS 是注入脚本到目标网站，让目标网站本身执行恶意代码
- CSRF 不在目标网站执行代码，而是诱骗用户对目标网站发请求

### 4.2 SQL 注入的原理及防范

**答**：

**原理**：

应用拼接 SQL 时没做转义，攻击者构造特殊输入改变 SQL 语义。

```java
// 错误示例
String sql = "SELECT * FROM users WHERE name = '" + name + "'";
// name 输入 admin' OR '1'='1
// 实际 SQL: SELECT * FROM users WHERE name = 'admin' OR '1'='1'
// 返回所有用户
```

**危害**：

- 数据泄露（脱库）
- 数据篡改
- 拖库 / 删库
- 提权（注入到 admin 表）

**防范**：

1. **参数化查询（最关键）**

```java
PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE name = ?");
ps.setString(1, name);
```

参数和 SQL 模板分开，DB 把参数当数据不当代码。

2. **ORM 框架**

MyBatis 用 `#{name}` 而不是 `${name}`：

- `#{}`：参数化查询，安全
- `${}`：字符串拼接，有注入风险（只用于表名、列名等动态部分）

3. **最小权限原则**

业务账号只给必要权限（SELECT/INSERT），不给 DDL / DROP。

4. **输入校验**

- 白名单：用户输入只允许合法字符
- 长度限制
- 特殊字符转义

5. **WAF**

Web Application Firewall 自动检测注入特征。

### 4.3 DDoS 攻击是什么？如何防范？

**答**：

**DDoS（Distributed Denial of Service）分布式拒绝服务**：

利用大量肉鸡同时向目标发送请求 / 流量，耗尽目标资源（带宽、CPU、连接数），让正常用户无法访问。

**常见类型**：

- **流量型**：UDP flood、ICMP flood，打满带宽
- **协议型**：SYN flood、ACK flood，打满 TCP 连接
- **应用层**：HTTP flood、慢连接（Slowloris），打满业务资源
- **DNS 反射放大**：利用 DNS 查询的小请求 + 大响应特性放大流量

**防范**：

1. **基础设施层**

- 高带宽 + 多机房 / 多线路
- BGP Anycast（同一 IP 多个节点）
- DDoS 高防 IP（云厂商提供，把流量先牵引到清洗集群）

2. **协议层**

- SYN Cookies
- 限制单 IP 连接数 / QPS
- 启用 TCP RST 攻击防护

3. **应用层**

- 验证码 / 人机识别
- 限流（IP / 用户 / 接口）
- 业务逻辑优化（耗时操作异步化）
- WAF 过滤异常请求特征

4. **CDN / 缓存**

- 静态内容走 CDN
- 边缘节点拦截恶意流量

**实战经验**：

- 中小规模（< 10G）：云防护就够
- 大规模（百 G+）：必须上专业高防服务
- 持续型：考虑切 IP、上 Anycast

### 4.4 JWT 的原理及安全性问题

**答**：

**JWT（JSON Web Token）**：

无状态的 token 认证方案。三段以 `.` 分隔：

```
Header.Payload.Signature
```

- **Header**：`{"alg":"HS256","typ":"JWT"}`，Base64URL 编码
- **Payload**：业务声明 + 标准字段（exp、iat、sub），Base64URL 编码
- **Signature**：HMAC(Header + "." + Payload, secret)

服务端拿同一个 secret 验签，验过则信任 payload。

**优点**：

- 无状态（服务端不存 token）
- 可水平扩展（任意节点都能验）
- 跨域 / 跨语言友好

**安全性问题**：

**1. 无法注销**

JWT 一旦签发，在过期前都有效。注销需要服务端维护黑名单（变成有状态）。

**2. Payload 可读**

只是 Base64 编码，没加密。**不能在 payload 放敏感信息**（密码、银行卡）。

**3. 算法替换攻击**

历史 CVE：把 `alg: HS256` 改成 `alg: none`，服务端如果接受 `none` 算法就无需 Signature。**实现要强制校验算法**。

**4. 密钥泄露 = 全员沦陷**

secret 一旦泄露，攻击者可以伪造任意用户的 token。

**5. 长期有效问题**

为了减少登录次数，token 有效期长，但被盗后影响时间也长。**短期 access_token + 长期 refresh_token** 方案：

- access_token 15 分钟过期（被盗影响小）
- refresh_token 7 天过期（只用于换新 access_token）
- 注销时把 refresh_token 加黑名单

**6. CSRF**

如果 JWT 存 cookie，仍可能 CSRF。存 localStorage 又有 XSS 风险。

**实践建议**：

- 用 HS256 / RS256 / ES256 强算法
- 强制校验 `alg`
- 短过期 + refresh 机制
- 关键操作（改密码、转账）二次验证
- 监控异常使用模式（IP 突变）

### 4.5 OAuth 2.0 的流程是什么？

**答**：

**OAuth 2.0**：授权框架。让第三方应用在不获取用户密码的前提下访问用户在其他平台的资源。

**四个角色**：

- **Resource Owner**：用户
- **Client**：第三方应用
- **Authorization Server**：授权服务器（如微信、GitHub）
- **Resource Server**：资源服务器

**四种授权模式**：

**1. 授权码模式（Authorization Code，最常用）**

```
1. 用户访问 Client，Client 重定向到授权服务器
   GET /authorize?response_type=code&client_id=X&redirect_uri=Y&scope=read
2. 用户在授权服务器登录 + 同意授权
3. 授权服务器重定向回 Client，带 code
   GET https://client.com/callback?code=abc123
4. Client 用 code 换 access_token（后端调用，不经过浏览器）
   POST /token?grant_type=authorization_code&code=abc&client_secret=...
5. 授权服务器返回 access_token (+ refresh_token)
6. Client 用 access_token 调资源服务器 API
```

适合：传统 Web 应用（有后端）。

**2. 简化模式（Implicit）**

直接在第 3 步返回 access_token（不要 code）。已被弃用，因为 token 暴露在 URL。

**3. 密码模式（Resource Owner Password Credentials）**

用户直接把账号密码给 Client，Client 拿密码去换 token。**只能用于高度信任的官方应用**。

**4. 客户端模式（Client Credentials）**

机器对机器认证，没有用户参与。Client 用 client_id + client_secret 换 token。

**access_token 和 refresh_token**：

- access_token 短期（小时级）
- refresh_token 长期（天 / 月级），用来换新 access_token
- access_token 失效时用 refresh_token 静默刷新，不打断用户

**OAuth 跟 OIDC（OpenID Connect）的关系**：OAuth 是授权（你能干什么），OIDC 是认证（你是谁）。OIDC 基于 OAuth 2.0 扩展，加了 id_token（含用户信息）。
