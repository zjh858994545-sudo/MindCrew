# 企业文档：监控、告警与故障处置

## 文档目标

本文档用于说明知识库问答系统上线后应该重点关注哪些指标、如何设置告警，以及出现故障后如何做初步分级和处置。

## 一、为什么需要监控和告警

知识库问答系统的稳定性不仅取决于应用本身，还依赖多个外部组件：

- 数据库
- 缓存
- 对象存储
- 向量数据库
- 大模型服务
- 外部搜索服务

任何一层异常，都可能表现为：

- 文档处理失败
- 问答超时
- 结果为空
- 来源缺失
- 页面可用但答案质量显著下降

## 二、建议重点监控的指标

### 应用层

- 接口成功率
- SSE 建连成功率
- 平均响应时间
- P95/P99 响应时间

### 检索层

- 向量检索耗时
- 关键词检索耗时
- 重排耗时
- 空结果比例

### 模型层

- Chat Model 调用成功率
- Embedding 调用成功率
- 外部搜索调用成功率

### 数据层

- MySQL 连接池使用率
- Redis 连接与内存使用
- Milvus 查询耗时
- MinIO 上传失败率

## 三、建议告警策略

### P1 告警

需要立即处理的场景：

- 登录不可用
- 问答主链不可用
- 文档上传全部失败
- 大量 5xx

### P2 告警

需要尽快处理但可短时观察的场景：

- 问答耗时显著升高
- 外部搜索大量失败
- 向量检索耗时异常
- 某些知识库处理任务连续失败

### P3 告警

用于趋势跟踪：

- 来源缺失率上升
- 兜底回答比例升高
- 文档导入量异常下降

## 四、常见故障分级建议

### Sev-1

用户核心功能不可用，影响广泛。

示例：

- 所有问答请求失败
- 登录全部失败
- 首页不可访问

### Sev-2

部分核心能力不可用，但可降级运行。

示例：

- 网页搜索不可用
- 部分知识库无法检索
- MCP 外部调用失败但内部问答正常

### Sev-3

局部问题或非核心异常。

示例：

- 导出功能异常
- 某类文档处理失败
- 某些统计页面数据不完整

## 五、初步处置流程

1. 确认告警是否真实
2. 判断影响范围
3. 检查最近发布和配置变更
4. 检查外部依赖状态
5. 决定降级还是回滚
6. 记录处置过程

## 六、典型故障排查路径

### 问答变慢

优先检查：

- 模型调用耗时
- 向量检索耗时
- 是否出现大量重试
- 是否有依赖组件 CPU/内存异常

### 回答频繁无来源

优先检查：

- 知识库是否为空
- 文档是否导入完成
- 来源组装逻辑是否异常
- Prompt 是否被错误裁剪

### 文档导入失败

优先检查：

- 文件格式是否支持
- 对象存储是否可写
- 文本抽取是否报错
- 向量数据库是否可写

## 七、值班建议

- 发布当日安排研发和运维同时在线
- 核心时段开启重点指标盯盘
- 对连续失败任务设置自动聚合告警，避免告警风暴

## 总结

监控和告警的目标不是“把所有异常都报警”，而是“在真正影响用户和业务之前尽早识别风险”。对 MindCrew 这类多依赖系统，跨层指标和统一排障入口尤为重要。
