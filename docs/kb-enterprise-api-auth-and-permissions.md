# 企业文档：API 鉴权与权限模型说明

## 文档目标

本文档描述企业系统在对外提供 API 时常见的鉴权与权限控制规则，适合作为中后台系统、开放平台和内部服务治理的参考资料。

## 一、鉴权目标

API 鉴权的核心目标通常包括：

- 确认调用方身份
- 校验请求是否合法
- 限制可访问资源范围
- 记录审计日志
- 降低凭据泄漏带来的影响

## 二、常见鉴权方式

### 1. 用户令牌

适用于浏览器端、移动端或代表用户发起的请求。

典型特征：

- 登录后签发访问令牌
- 有明确过期时间
- 通常通过 `Authorization: Bearer <token>` 传递
- 需要配合刷新令牌或重新登录机制

### 2. 服务令牌

适用于服务之间调用或自动化任务调用。

典型特征：

- 与系统用户身份解耦
- 权限更适合按应用或环境维度控制
- 常配合 IP 白名单、签名或双向证书增强安全性

### 3. 请求签名

适用于高敏感接口或开放平台接口。

常见做法：

- 请求中带时间戳和随机串
- 关键参数按固定顺序拼接
- 使用应用密钥计算签名
- 服务端验签并校验重放攻击窗口

## 三、推荐的权限模型

### 1. 用户、角色、权限点三层模型

推荐将权限拆成三层：

- 用户：谁在调用
- 角色：用户属于什么身份
- 权限点：允许执行什么动作

示例角色：

- 超级管理员
- 知识库管理员
- 文档维护者
- 普通成员
- 只读访客

示例权限点：

- `kb:read`
- `kb:upload`
- `kb:delete`
- `chat:stream`
- `mcp:invoke`
- `admin:config:update`

### 2. 资源级权限

如果系统支持团队空间或多知识库隔离，权限还应支持资源级约束，例如：

- 只能访问自己所在团队的知识库
- 只能管理自己创建的文档
- 只能调用被授权的 MCP 工具

## 四、接口鉴权建议

### Header 约定

推荐至少统一以下请求头：

- `Authorization`
- `X-Request-Id`
- `X-Client-Id`
- `X-Timestamp`

如果需要审计和排障，`X-Request-Id` 尤其重要。

### Token 生命周期

建议遵循以下原则：

- 访问令牌有效期较短
- 刷新令牌有效期较长
- 支持主动失效
- 支持多端登录策略

### 异常返回

建议区分以下场景：

- `401`：未登录、令牌失效、签名错误
- `403`：已登录但无权限
- `429`：请求频率过高

## 五、MCP 场景下的权限控制

如果系统对外暴露 MCP 工具，建议额外控制：

- 哪些客户端可访问 `/mcp`
- 哪些工具允许被调用
- 哪些工具需要更高权限
- 工具调用是否计入审计日志

示例：

- `doc_search` 可对普通成员开放
- `store_memory` 需要登录用户上下文
- `admin` 类工具仅管理员可用

## 六、审计建议

对以下信息建议记录审计日志：

- 调用时间
- 用户或客户端标识
- 目标接口或工具名
- 请求来源 IP
- 结果状态
- 耗时
- 失败原因

审计日志的价值在于：

- 安全追踪
- 合规检查
- 问题回溯
- 使用分析

## 七、企业实践建议

- 不要把管理员权限直接绑定到所有后台账号
- 不要长期使用永久有效令牌
- 不要在日志中明文打印完整 token
- 对高风险接口增加二次确认或更严格认证
- 对外集成接口配置调用配额和速率限制

## 八、总结

一个成熟的 API 鉴权方案，不只是“能不能访问”的问题，而是“谁可以在什么条件下访问哪些资源，并且全程可审计”的问题。对企业系统而言，身份、权限、审计三者需要同时成立，鉴权体系才算完整。
