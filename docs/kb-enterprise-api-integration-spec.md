# 企业文档：API 集成规范与调用约定

## 文档目标

本文档用于定义企业内部或对外开放 API 的通用集成规范，涵盖请求格式、分页过滤、幂等性、错误码和重试建议。

## 一、基础约定

### 1. 协议与路径

推荐约定：

- 使用 HTTPS
- 路径采用名词化设计
- 版本号放在路径中，例如 `/api/v2/...`

示例：

- `/api/v2/chat/stream`
- `/api/v2/chat/conversations`
- `/api/mcp/tools`

### 2. 数据格式

推荐默认使用 JSON：

- 请求体：JSON
- 返回体：JSON
- 流式接口除外，例如 SSE

## 二、请求规范

### 1. 查询参数

适用于列表查询、筛选和分页请求。建议统一：

- `current`：当前页码
- `size`：分页大小
- `keyword`：关键词
- `sortBy`：排序字段
- `order`：排序方向

### 2. Header 规范

建议统一：

- `Authorization`
- `Content-Type`
- `X-Request-Id`

### 3. 时间与日期格式

建议统一使用：

- ISO 8601
- 或明确指定时区的 `yyyy-MM-dd HH:mm:ss`

关键在于所有接口保持一致。

## 三、响应规范

### 1. 统一响应结构

推荐保持：

- `code`
- `message`
- `data`

示例：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "id": 123
  }
}
```

### 2. 列表响应

分页接口建议返回：

- 当前页
- 页大小
- 总记录数
- 当前页数据列表

## 四、幂等性设计

### 1. 哪些接口需要幂等

建议以下操作具备幂等保护：

- 创建订单
- 创建任务
- 触发审批
- 重复提交文件上传记录

### 2. 常见实现方式

- 客户端传 `Idempotency-Key`
- 服务端记录请求指纹
- 在有效窗口内识别重复请求

## 五、错误码与失败处理

### 常见状态建议

- `200`：成功
- `400`：参数错误
- `401`：未认证
- `403`：无权限
- `404`：资源不存在
- `409`：状态冲突
- `429`：频率限制
- `500`：服务内部错误

### 失败返回建议

除了状态码外，`message` 应尽量清晰，例如：

- 参数格式错误
- 当前状态不允许执行该操作
- 资源已被删除
- 请求频率超限

## 六、重试建议

### 可自动重试的场景

- 网络抖动
- 网关超时
- 临时性 `5xx`

### 不建议自动重试的场景

- 参数错误
- 权限错误
- 业务冲突

### 重试策略

推荐使用：

- 指数退避
- 最大重试次数限制
- 幂等键保护

## 七、异步任务接口建议

对于耗时任务，如文档解析、批量导入、索引构建，建议采用异步任务模式：

1. 提交任务
2. 返回任务 ID
3. 客户端轮询或订阅任务状态

状态建议包括：

- `pending`
- `running`
- `success`
- `failed`

## 八、Webhook 与回调

如果系统支持回调通知，建议：

- 使用签名校验
- 支持重放保护
- 定义明确重试策略
- 明确回调幂等规则

## 九、总结

企业 API 的可用性不只取决于“接口能不能调用”，还取决于它是否稳定、清晰、可重试、可审计、可扩展。统一的 API 规范能显著降低接入成本和后期维护成本。
