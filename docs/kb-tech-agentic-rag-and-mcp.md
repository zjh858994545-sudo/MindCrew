# MindCrew 技术文档：Agentic RAG 与 MCP 能力说明

## 文档目标

本文档解释 MindCrew 中 Agentic RAG 和 MCP 的职责划分、内部调用方式、对外暴露方式，以及它们在问答系统中的协作关系。

## 一、什么是 Agentic RAG

在 MindCrew 中，Agentic RAG 指系统不再只是“拿到问题后固定检索再固定生成”，而是会根据问题类型动态决定：

- 是否需要改写查询
- 应该用哪些工具
- 是否需要知识库检索
- 是否需要关键词精查
- 是否需要网页搜索
- 是否需要读取或写入长期记忆

这种方式让系统更像一个受约束的任务执行器，而不是单纯的模板流水线。

## 二、MindCrew 中的主要工具

### 1. `doc_search`

用途：

- 执行语义向量检索
- 适合概念解释、相似表达、模糊提问

输入示例：

- 查询词
- topK
- 可选的知识库范围

### 2. `keyword_search`

用途：

- 执行关键词检索
- 适合查编号、专有名词、配置项、固定短语

### 3. `web_search`

用途：

- 补充外部网页信息
- 适合实时更新、外部公告、版本变化、公共资料

### 4. `recall_memory`

用途：

- 读取用户长期记忆
- 适合追问、个性化回答、延续上下文

### 5. `store_memory`

用途：

- 将用户明确表达的长期偏好写入 Redis
- 例如角色、称呼、表达偏好、关注主题

## 三、问题进入系统后的决策过程

系统会先做问题路由。常见意图包括：

- 普通知识查询
- 精确检索
- 实时信息查询
- 复合问题
- 追问问题

不同意图会触发不同工具组合。例如：

- 一般查询：`doc_search + keyword_search`
- 实时问题：`doc_search + keyword_search + web_search`
- 追问问题：`recall_memory + doc_search`

## 四、内部调用与外部调用的区别

### 内部调用

MindCrew 应用内部并不会通过 MCP 协议去回调自己，而是：

- 直接注入工具对应的 Spring Bean
- 在 `MindCrewAgent` 中直接调用工具方法

优点是：

- 调用链更短
- 性能更稳定
- 更容易调试

### 外部调用

对外部 Agent、脚本或支持 MCP 的客户端，MindCrew 会通过 MCP Server 对外暴露工具能力。外部客户端可以通过 HTTP/SSE 发现工具并调用。

这意味着：

- 内部是“本地函数调用”
- 外部是“标准 MCP 工具调用”

两者共用同一批工具实现，但走不同的接入路径。

## 五、为什么要同时保留 Agentic RAG 和 MCP

### Agentic RAG 解决内部问答编排问题

它负责：

- 怎么决定下一步动作
- 怎么组织检索与生成
- 怎么把多路来源合成最终答案

### MCP 解决能力复用问题

它负责：

- 如何把工具能力开放给外部
- 如何让其他 Agent 或自动化流程复用知识库能力
- 如何实现工具发现、标准化输入输出和调用记录

## 六、典型调用链示例

### 场景一：技术规范问答

问题：

“权限系统里角色继承怎么配置？”

可能链路：

1. 判断为普通知识库查询
2. 改写查询
3. 调用 `doc_search`
4. 调用 `keyword_search`
5. 融合重排
6. 输出带来源的答案

### 场景二：版本变化查询

问题：

“最新发布里鉴权方式有变化吗？”

可能链路：

1. 判断为实时问题
2. 调用 `doc_search`
3. 调用 `keyword_search`
4. 调用 `web_search`
5. 统一编号来源
6. 给出结论并附上网页链接

### 场景三：追问

问题：

“继续说刚才那个权限模型的限制”

可能链路：

1. 判断为追问
2. 调用 `recall_memory`
3. 结合对话历史和知识库继续回答

## 七、MCP 接入时需要注意的问题

- `/mcp` 接口是否要求鉴权
- 工具输入输出是否稳定
- 工具异常是否会导致整个请求失败
- 外部搜索是否已配置
- 工具调用结果是否需要审计

## 八、工程实践建议

- 内部业务优先直接复用工具 Bean，不要为内部场景绕一层 MCP
- 外部集成统一走 MCP，减少私有协议
- 工具命名保持稳定，避免外部客户端适配成本过高
- 所有工具返回结构化结果，避免返回不可解析文本

## 总结

在 MindCrew 中，Agentic RAG 是“如何编排问题求解过程”，MCP 是“如何开放工具能力给外部调用”。两者不是互相替代关系，而是内部编排与外部集成的两层能力。
